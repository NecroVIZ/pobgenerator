# Ревью проекта: генератор билдов PoE1 (Phase 0 — интеграция с Path of Building)

> Документ для передачи стороннему ревьюеру (в т.ч. другой нейросети). Самодостаточный
> контекст: цель, ограничения, архитектура, ключевые технические решения, состояние
> тестов, риски и вопросы. Канонические источники в репозитории: `DESIGN-v2.md`
> (консолидированная спека), `REVIEW-of-DESIGN-v2.md`, `SPIKES.md`.
> Отчёт о спайках B/D/E и joint: `WORK-REPORT.md`.

## 1. Что это и зачем
Цель — генератор **готовых жизнеспособных билдов** для Path of Exile 1, ориентированный на
поиск **необычных, но сильных комбинаций** в текущих данных игры (полные сборки: дерево
пассивок, гемы, снаряжение, флаконы, бандиты/пантеон). PoB используется как **оракул правил
и калькулятор** (DPS/EHP/резисты).

Сейчас завершён **Phase 0** — надёжная интеграция с PoB как headless-движком + фундамент
(модель билда, контракт-валидатор, gate данных).

## 2. Стратегия генерации — correctness-first (ВАЖНО, частая путаница)
Per `DESIGN-v2.md` §1.2–1.3, решения **A1/D1**:

> «архитектура остаётся **correctness-first** (CP-SAT + fixpoint как ядро), **НЕ** разворачивается
> на seed-and-mutate.»

Ядро (DESIGN-v2 §2, §5):
```
скелет (skill × ascendancy × оси новизны) → skeleton-gate (дёшево, без PoB) →
реализатор [ дерево: target-set нотаблов → Dijkstra-stitching → point-budget pruning (D14)
            + снаряжение: cost-constrained CP-SAT (D15)
            + гемы ] → структурный gate → оценка в PoB → локальная оптимизация → ранжирование
```

**seed-and-mutate как способ генерации — отклонён** (§1.3): наследует ошибки семян
(PoB-config-abuse реальных билдов), а discovery живёт в формальной метрике `unusualness` (§8),
а не в ширине перебора; CP-SAT решает «craftable + budget + constraints», чего мутация не умеет.
Оставлен лишь как **опциональный breadth-search слой в Phase 2**, если throughput критически мал.

Не путать со **локальным хилл-клаймбом** (§7, Phase 3): он «мутирует» (своп узла/мода/гема), но
применяется к **уже корректно собранному** реализатором билду — это оптимизация, не генерация.

Цель «новая мета» осознанно **понижена** (A2) до «необычное+сильное в текущих данных»: прогноз
популярности будущих лиг недостижим до появления данных в PoB.

## 3. Окружение и ограничения (контекст реализации)
- Windows, **только Python 3.11**. Нет `git`, системного `luajit`, установленного PoB, нет
  пакетных менеджеров. Сеть ограничена (CDN релизов рвётся; помог `codeload.github.com`).
- PoB используется как **исходники** (vendored, **v2.65.0**, дерево **3.28**), каталог `vendor/`
  в `.gitignore`.
- LuaJIT — не системный, а **встроенный в Python через `lupa` (`lupa.luajit21`)**.

## 5. Ключевые технические решения и хаки (главное для ревью)
1. **Встроенный LuaJIT через lupa** (системного нет). Рантайм с `encoding=None` — Lua-строки
   ходят как `bytes` (нужно для бинарных данных); utf-8 на границах Python↔Lua.
2. **Нативные C-модули PoB не используются** (`lua-utf8`, `lzip`, `lcurl`): ABI-несовместимы со
   встроенным LuaJIT (отдельный `lua51.dll`). Заменены pure-Lua/Python-шимами; `lua-utf8` → фолбэк
   на `string` (ASCII-достаточно для разбора билдов/статов).
3. **`Inflate`/`Deflate` через Python `zlib`** подменяют заглушки PoB — иначе молча не грузятся
   сжатые data-файлы (таймлесс-джевелы) и **тихо ломается дерево**.
4. **`GetScriptPath` оверрайдится** на абсолютный путь к `src` — иначе `scriptPath.."/Data/..."`
   даёт битый путь и снова тихо рушит загрузку данных.
5. **`NewFileSearch` реализован заново** (в HeadlessWrapper — пустышка): Python-глоб + handle-
   таблица на Lua (`:GetFileName()/:GetFileModifiedTime()/:NextFile()`). Без него не грузятся
   **split-файлы** Glorious Vanity (`GloriousVanity.zip.part0..4`).
6. **Изоляция через подпроцесс**: несколько экземпляров PoB в одном процессе дают фатальный краш
   LuaJIT (SEH `0xe24c4a02`). Решение — процесс на оценку.
7. **Маршалинг Lua→Python**: `lupa.lua_type()` на этом рантайме возвращает `None`, поэтому
   table-proxy детектится по модулю `lupa` + `.items()`; таблица с ключами 1..n → list, иначе dict.

## 6. build-model (принцип losslessness)
- **Типизированы** части, которые мутирует генератор: `BuildMeta` (level/класс/аскенд),
  `Tree`/`Spec` (узлы как `list[int]`, `mastery_effects` как `dict[int,int]`),
  `Skills`/`SkillSet`/`SkillGroup`/`Gem`.
- **Lossless-passthrough** остального: верхнеуровневые `Import/Party/Notes/Calcs/TreeView/Items/
  Config` — вербатим в исходном порядке; нетипизированные дочерние узлы (критично — `Spec/URL/
  Sockets/Overrides` с джевелами) — тоже сырым XML.
- Все атрибуты хранятся **сырыми строками**, типобезопасный доступ — через свойства (без потери
  формата `true`/`false`/`nil`).
- Контракт: `from_xml(real).to_xml()` → PoB считает статы **идентично** (проверено, отклонение < 1e-6).

## 7. Контракт-валидатор (PoB как оракул)
- **ERROR** (фитнесу нельзя доверять): не загрузился; нет активного скилла; гем не распознан
  (`gemInstance.errMsg` **или** отсутствие `gemData`/`grantedEffect`); мастери аллоцирована без
  выбранного эффекта.
- **WARNING**: неразобранный остаток мода предмета (`modLine.extra`).
- Принимает XML-строку, `None` (пустой билд) или объект `PobBuild`.

## 8. Data-gate (вместо RePoE)
Источник истины — **данные самого PoB**, не RePoE. Gate проверяет: версию движка (`2.65.0`),
версию дерева (`3_28`), число гемов (858), разрешимость опорных активных/саппорт-гемов, загрузку
**всех 6** типов таймлесс-данных, парсинг образцовых модов без остатка.
Текущий статус: `pob=2.65.0 tree=3_28 gems=858 timeless=6/6 mods=6/6 ok=True`.

## 9. Состояние тестов — 26/26 зелёные, **exit 0** (без SEH-краша)
> Ранее «27 passed» сопровождалось фатальным `0xe24c4a02` (SEH LuaJIT) на teardown
> in-process `PobHeadless` → pytest exit 1 (в CI = failed). Исправлено: **в pytest-сеансе
> больше нет in-process `PobHeadless`** — все тесты гоняют PoB только через изолированный
> процесс (`evaluator`). In-process путь допустим лишь в одноразовых `scripts/`.

- `test_pobcode` — кодек (round-trip lossless).
- `test_headless` (2, subprocess) — версия движка + вменяемые дефолты пустого билда.
- `test_roundtrip` — детерминизм XML→headless→XML'→headless + lossless-кодек + сверка `PlayerStat`.
- `test_validator` (3) — пустой билд валиден; реальный проходит контракт; сломанный гем ловится.
- `test_model` (6) — разбор; сохранение структуры; **round-trip дублей тегов**; мутации;
  **идентичность статов через PoB**.
- `test_datagate` (3) — gate проходит + пороги + покрытие `NewFileSearch` (5 частей GV, nil на отсутствии).
- `test_gold` — **золотой тест**: headless == GUI на реальном билде (Marauder/Chieftain Blade
  Vortex), 98/98 статов, расхождение 0.000%.

## 10. Известные ограничения / риски
- Проверено по сути на **одном реальном билде** (golden). Нет широкого корпуса архетипов
  (минионы, тоталемы, триггеры, конверсия, аилменты).
- `lua-utf8` → ASCII-фолбэк: возможны проблемы на нелатинских данных (для расчётов, вероятно,
  неважно, но не доказано).
- Привязка к конкретной версии PoB/дерева (2.65.0 / 3.28); апгрейд не автоматизирован (на это
  нацелен data-gate).
- Изоляция «процесс на оценку» корректна, но **дорого по производительности** для массовой
  генерации (Phase 2: тысячи оценок). По дизайну (D32) — multiprocess + batch + checkpoint-resume.
- Сам реализатор (скелеты, дерево D14, CP-SAT по снаряжению D15, fixpoint D1, gate'ы, fitness)
  **ещё не написан** — это Phase 1+.

## 11. Что дальше по дизайну (Phase 1, DESIGN-v2 §11)
**Реализатор + два контрастных билда** (D3):
- **1a (лёгкий):** Templar/Inquisitor, Arc-семейство, spell+crit+lightning, Determination+Grace+suppression.
- **1b (жёсткий):** attribute-stacking (нагружает fixpoint) ИЛИ конверсия (нагружает gate резистов).
- Критерий: оба собираются, на 1b fixpoint сошёлся (D1), gate'и прошли, round-trip ±1%.

Первый кирпич — **скелет + реализатор дерева** (target-set нотаблов → Dijkstra-stitching →
point-budget pruning), затем CP-SAT по снаряжению и локальный хилл-клаймб.

## 12. Вопросы к ревьюеру
1. Корректен ли фундамент интеграции с PoB? Есть ли надёжнее/дешевле, чем lupa + подпроцесс на
   оценку (пул прогретых воркеров, shared memory, батчинг)?
2. Достаточно ли контракт-валидатора как «оракула легальности», или нужны доп. проверки
   (требования атрибутов/уровня гемов, лимиты сокетов/линков, легальность аффиксов, бандиты/пантеон)?
3. Реализатор correctness-first (CP-SAT + fixpoint + декомпозиция дерева D14): где наибольший риск
   нерешаемости/комбинаторного взрыва и «тихо-неверных» билдов? Альтернативы (MILP, иное)?
4. Не упускаем ли механики, которые PoB считает неточно/не считает (специфические триггеры, кастомные
   моды), и как это валидировать без RePoE?
5. Производительность Phase 2: как масштабировать до десятков тысяч оценок (распараллеливание,
   кеш по хешу билда, инкрементальный пересчёт)?
6. Метрика `unusualness` (curated + discovery, §8): достаточно ли она «честна», нет ли вырождения
   (degenerate-guards D34)?

## 13. Правки по итогам внешнего ревью (сделано в Phase 0)
- **B1 (критично) — процесс-краш на выходе:** убран in-process `PobHeadless` из pytest;
  `test_headless` переписан на subprocess; NFS-покрытие перенесено в `datagate`. Теперь exit 0.
- **B2 (критично) — round-trip дублей тегов в `model.py`:** dict с `#N`-ключами заменён на
  упорядоченную раскладку `layout` + список `passthrough`; добавлен юнит-тест на дубли.
- **B3 — shebang-срез:** только `#@`/`#!`, а не любой `#`-первый-символ.
- **B4 — `recalc` конвергенция:** вместо early-break — ожидание 2 чистых кадров подряд (до 20),
  устраняет тихую недооценку статов на тяжёлых билдах.
- **B5 — `worker.py`:** `BaseException` → `Exception` (не глотаем KeyboardInterrupt/SystemExit).
- **B6/B7 — косметика:** комментарий про list-vs-dict с дырами; точнее сообщение об ошибке `pobcode`.
