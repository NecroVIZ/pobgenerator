#### Ответы на Q1–Q5

**Q1. ML-primary vs greedy как финал?**

**Согласен с ML-primary как стратегической ставкой автора**, но с жёстким условием: ML — не
замена PoB-оракулу, а **learned prior** над пространством деревьев/таргет-сетов. PoB остаётся
likelihood (DPS/EHP/легальность). Эвристики (sd-теги, чистый CP-SAT-maximize) эмпирически мертвы —
это мы доказали, не гипотеза.

**Контраргумент (не для отката, а для scope):** greedy+hill-climb на дереве — дешёвый **нижний
занавес** (lower bound). Его надо добить до ~40–50% overlap **до** большого ML-инвеста (~3–5 дней),
иначе ML сравнивают с мёртвым sd-baseline (5%), а не с лучшей эвристикой (34%). Автор выбрал
ML-primary — ок, но **gate = max(greedy-hill-climb, ML)**, не «только ML».

**Q2. ML-архитектура для дерева?**

Предлагаю **двухуровневую** схему (не одну модель на всё):

1. **Уровень 1 — target-set predictor (3A-2):** gradient boosting (LightGBM/CatBoost) или небольшой
   MLP над **axis-descriptor** → scores по ~200–400 кандидат-notables (prefilter sd или PoB cheap
   probe). Выход: top-K notables → **существующий Dijkstra** (oracle 96–100% — не переобучать).
   *Почему не GNN сразу:* граф дерева огромен (~1.5k узлов), разметка «полное дерево» дорога,
   а stitching уже решён. GNN оправдан на **уровне 2**, не сейчас.

2. **Уровень 2 — tree prior (3C-3), позже:** GNN или autoregressive notable-selector с
   budget-constraint (SET transformer / pointer network), если уровень 1 упирается в потолок.

**Минимальный ML-эксперимент (1–2 недели):** уровень 1 только. Train: `(axis_descriptor, notable_id) →
binary «в эталоне?» или regression «marginal DPS если добавить» (из PoB offline на корпусе).
Eval: overlap@K + **DPS% после Dijkstra** на holdout билдах (те же 6 + 5 holdout из ninja).

**Q3. Axis-descriptor как числовой вход?**

Предлагаю **три слоя** (конкатенация):

| Слой | Источник | Размерность |
|---|---|---|
| **A. Скелет** | class, ascendancy, main_skill, skill_tags (из PoB gem data) | ~50–100 one-hot |
| **B. Оси §4.1** | damage_type (ele/phys/chaos), delivery (hit/dot/minion), defense_axis (life/es/evasion/armour), mechanic_flags (conversion, crit, ailment) | ~20–40 binary (ручная схема + валидатор) |
| **C. PoB-сигналы** | marginal DPS top-20 notables **на minimal tree** (offline, раз на корпус) | 20 floats |

Слой C — мост между «чистым ML» и PoB: модель учит **корреляции** marginals↔эталон, не заменяет PoB.
Слой B — из DESIGN §4.1, нужно формализовать в `axis.py` (ещё нет).

**Разметка axis для ninja-корпуса:** автоматически из PoB XML (class, skill, keystones allocated) +
rule-based теги (ignite если allocated notables содержат ignite%, и т.д.). Ручная — только для
`builds/` как gold.

**Q4. Diversity / anti-meta loss?**

Для **генератора необычных билдов** pure imitation = смерть цели. Предлагаю:

1. **Training:** contrastive — positive = эталонное дерево билда, negative = случайное дерево той же
   класс/бюджет или популярный ninja-билд с тем же скиллом.
2. **Inference:** `score(n) = ML_prior(n) + λ·PoB_marginal(n) − μ·popularity(n)` где popularity =
   частота notable в ninja-корпусе для этого skill.
3. **Fitness (Phase 2):** явный `unusualness` term из DESIGN §8 — ML не оптимизирует его напрямую,
   но **отсекает** popular-flag-sets через μ.

Без μ и contrastive ML воспроизведёт мету — согласен с твоим опасением.

**Q5. Greedy-baseline как обязательный gate?**

#### Что блокирует push
**PAT (Personal Access Token) от автора.** dulwich push по HTTPS требует `https://<PAT>@github.com/...`.
Без системного git нет credential-helper'а, поэтому PAT — единственный путь.

**Инструкция автору (как дать PAT, не раскрывая его мне/в чате):**
1. GitHub → Settings → Developer settings → Personal access tokens → Tokens (classic) →
   **Generate new token**, scope = **`repo`** (полный доступ к репо), expiry = на ваше усмотрение.
2. Скопировать PAT (`ghp_...`).
3. Запустить push с PAT в URL (ВАМ, не мне — PAT не должен попадать в чат/логи):
   ```
   cd C:\Users\KZ\Desktop\s
   python scripts\git_backup.py push --remote origin --url "https://<ВАШ_PAT>@github.com/NecroVIZ/pobgenerator.git"
   ```
4. После push — **отозвать PAT** (GitHub → token → Revoke), или сократить expiry. PAT в истории
   shell/логов = риск; разовый push + revoke = безопасно.

**Альтернатива (если не хотите PAT в URL):** установить системный git
(https://git-scm.com/download/win → PortableGit, распаковать в `C:\git`, добавить в PATH), тогда
`git push` с credential-helper'ом. Дороже setup'а, но безопаснее для PAT.

dulwich создал branch **`master`** (не `main`). GitHub по умолчанию создаёт `main`. При первом push:
- либо GitHub примет `master` (создаст ветку master; default останется main — репо «пустое» до push),
- либо нужно переименовать. Для бэкапа master-vs-main **не критично** (обе ветки в репо).
