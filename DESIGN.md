# Генератор билдов для Path of Exile 1 — Дизайн-документ

> Статус: черновик v0.1 (итерируем)
> Цель проекта: генерировать **готовые, жизнеспособные билды** для PoE1, ориентируясь на
> поиск **новой меты и необычных комбинаций**. Не «идеи-концепты», а полные сборки:
> класс/аскенденси, скиллы и линки, дерево пассивик, снаряжение с аффиксами, джуэлы,
> фласки, бандиты/пантеон.

---

## 1. Главная проблема и принцип решения

Сгенерировать билд, который одновременно:
1. **Полный** — все компоненты заполнены и согласованы между собой.
2. **Жизнеспособный** — честные DPS/выживаемость, капнутые резисты, хватает маны/жизни.
3. **Реалистичный** — снаряжение реально craftable/доступно в игре (а не «PoB-warrior»).
4. **Новый** — комбинация недопредставлена в текущей мете.

Пространство поиска астрономическое и сильно связанное (скилл-спелл требует спелл-скейлинг,
конверсия требует согласованного дерева и шмота и т.д.). Поэтому **не ищем в сыром
пространстве**, а используем многослойную генерацию:

> **Ключевая идея: ищем в пространстве «скелетов билда», а затем детерминированно/эвристически
> достраиваем скелет до полной сборки и проверяем в PoB.**

«Скелет» = (аскенданси, главный скилл + набор сапортов, ядро-механика/тема скейлинга,
тема защиты). Это маленькое осмысленное пространство. По скелету:
- дерево достраивается как направленный поиск пути по тегам скейлинга и защиты;
- шмот — как constraint-fill реалистичными аффиксами под нужные теги;
- затем PoB оценивает.

---

## 2. Конвейер (pipeline)

```
[1] Генерация скелетов  ->  [2] Реализатор (скелет -> полная сборка)
        |                              |
   (инъекция новизны)            (дерево + шмот + гемы)
        v                              v
[5] Ранжирование по новизне  <-  [3] Оценка в PoB (DPS/EHP/валидность)
                                       |
                                       v
                                [4] Оптимизация (локальный поиск)
```

1. **Генерация скелетов** — кандидаты (skill × ascendancy × механика). Здесь же инъекция новизны.
2. **Реализатор** — превращает скелет в конкретный полный билд (самый трудоёмкий слой).
3. **Оценка в PoB** — headless-прогон: DPS, EHP, max hit, резисты, устойчивость ресурсов.
4. **Оптимизация** — локальный поиск/хилл-клайминг по дереву и шмоту вокруг кандидата.
5. **Ранжирование по новизне** — сверка с метой (poe.ninja): сильное + редкое = «новая мета».

---

## 3. Модель данных (внутренняя схема билда)

Единый internal-формат (pydantic/dataclasses), от которого есть сериализаторы в/из PoB.

```
Build {
  class, ascendancy, level (target),
  skills: [ SkillGroup { main_gem, supports[], slot } ],
  tree: { allocated_node_ids[], masteries[], jewel_sockets[] },
  gear: { slot -> Item },
  jewels: [ Jewel ],          # обычные / abyss / cluster
  flasks: [ Flask x5 ],
  bandits, pantheon,
  config: { enemy_level, boss, active_buffs, ... }   # конфиг расчёта PoB
}

Item {
  base, item_level, influences[], rarity,
  affixes: [ Affix { mod_id, tier, values[], group } ],
  corrupted, enchant
}
```

Этот формат — «лингва франка» проекта. Все генераторы пишут в него, PoB-сериализатор читает.

## 4. Генерация скелетов (движок новизны)

- Энумерация пар `(skill × ascendancy)` с фильтром механической совместимости по тегам.
- **Инъекция новизны:** вес в сторону низкопопулярных пар (по poe.ninja) и в сторону
  «правило-меняющих» уников как якорей механики (конверсии, «X становится Y», кейстоуны).
- **LLM-слой в контуре (решено):** LLM генерирует гипотезы-скелеты, но они ВСЕГДА проходят
  через реализатор + PoB — LLM не выдаёт цифры, только идеи под валидацию. Работает в паре
  с data-driven поиском: LLM расширяет пространство гипотез, данные/PoB его фильтруют.

### 4.1 Тема скейлинга = точка в ортогональном пространстве осей

Важно: «архетип» — это НЕ один ярлык, а комбинация независимых осей. Они ортогональны и
комбинируются (например, «спелл + хит + молния + крит» = крит Arc; «спелл + DoT + огонь
+ ignite» = ignite-билд). Оси:

| Ось | Значения |
|-----|----------|
| Способ доставки | спелл / атака оружием / миньоны / тотемы / трапы / мины / тригеры |
| Применение урона | хит (мгновенный) vs DoT (урон во времени) |
| Тип урона | физ / огонь / холод / молния / хаос (+ конверсии) |
| Эйлменты | ignite, poison, bleed (DoT-эйлменты) / shock, freeze, chill (не-урон) |
| Крит | крит vs не-крит |

> Следствие для дизайна: «скелет» билда фиксирует точку в этом пространстве, а реализатор
> и теги скейлинга работают именно с набором координат, а не с одним именем архетипа.
> Поэтому «спелл-кастер» и «DoT» — НЕ взаимоисключающие выборы.

## 5. Реализатор: скелет -> полная сборка (ядро инженерии)

### 5.1 Дерево пассивик
- Старт у стартовой точки класса.
- Взвешенный поиск пути (эвристика Steiner-tree): тянемся к нотаблам/кластерам, чьи статы
  совпадают с темой скейлинга + защита.
- Бюджет очков = по целевому уровню (например, 90–95) + квестовые очки.
- Учитываем мастери и сокеты под джуэлы.

### 5.2 Снаряжение
- Для каждого слота: выбор базы (по требуемым модам и требованиям скилла), затем заполнение
  аффиксов из **реальных пулов модов** (привязка к базе, группам модов, ilvl, инфлюенсам).
- Жёсткие ограничения: резисты капнуты, требования по атрибутам выполнены.
- **Уровень реалистичности (решено): Craftable.** Аффиксы берём ТОЛЬКО из реальных пулов
  модов (с учётом групп, ilvl, инфлюенсов) — то есть предмет в принципе может существовать
  в игре. БЕЗ учёта цены/доступности на старте (это отдельный, более поздний слой через ninja).
  Это отсекает «PoB-warrior» сборки, но не усложняет MVP экономикой.

### 5.3 Гемы / ауры / линки
- Набор сапортов под тип скилла из справочника «хорошие сапорты по тегам».
- Ауры/утилити (Determination/Grace/Defiance Banner/Herald-ы и т.п.) под тему защиты/урона.

> Реализатор — это экспертная система/эвристический слой. Самая большая часть работы, но
> именно он даёт «готовые билды», а не концепты.

## 6. Оценка через Path of Building (headless)

PoB — ядро расчётов. Нужен headless-прогон.

**Решение: гибрид A+B.**
- **Контракт = нативный XML-формат PoB** (то, во что декодируется import-code): документирован,
  читаем, отлаживаем, совпадает с форматом игры/комьюнити. Наш build-model <-> PoB XML.
- **Запуск = один долгоживущий headless-процесс luajit**, который держим «тёплым» и дёргаем
  `Modules/Calcs.lua` много раз (со стабом UI-слоя SimpleGraphic), а не перезапускаем PoB
  на каждый билд.
- Так берём корректность/отлаживаемость контракта (A) и производительность массового
  поиска (B). Вариант «subprocess на каждый билд» отвергнут — не выдержит тысячи прогонов.

Рассмотренные альтернативы:
- A в чистом виде (subprocess на билд) — слишком медленно для поиска.
- B в чистом виде (конструировать внутренние таблицы PoB напрямую) — хрупко и трудно отлаживать.

Fitness-вектор:
- Эффективный DPS (по PoB, с учётом конфигурации боя).
- EHP / max hit / восстановление.
- Жёсткие ограничения: резисты капнуты, мана/жизнь устойчивы, требования выполнены.

Производительность: агрессивное кэширование, батчи, дедуп одинаковых сборок.

---

## 7. Оптимизация

- На хорошем скелете — дешёвый локальный поиск: своп узла дерева у границы, замена одного
  мода/база в слоте, замена сапорт-гема; принимаем, если улучшает по Парето (DPS vs EHP).
- Полноценная генетика — позже, если локального хилл-клайминга мало.

---

## 8. Оценка «новизны» / «новой меты»

- Снапшоты билдов с poe.ninja -> частоты `(skill, ascendancy, ключевые уники, кейстоуны)`.
- `novelty = редкость комбинации`.
- **«Кандидат в новую мету» = высокий перформанс в PoB И низкая текущая популярность.** Это
  и есть флагманский вывод генератора.

---

## 9. Источники данных

- **Единый источник = data-экспорт самого PoB (решено).** Раз PoB — ядро расчётов, берём
  гемы/моды/базы/дерево/уники из его Lua-данных. Это гарантирует, что данные генерации и
  данные расчёта НЕ рассинхронятся. RePoE/официальный экспорт — только как сверка/дополнение.
- **poe.ninja API**: популярность билдов (база «обычного») + экономика (цены/доступность).

Кэшируем локально с **жёсткой привязкой к версии PoB/патча** (фиксируем хэш версии данных).

---

## 10. Стек

- **Ядро:** Python.
- `networkx` — пути по дереву; `pydantic` — модели; `httpx/requests` — poe.ninja.
- **PoB:** Lua через subprocess (luajit) с headless-скриптом, либо обёртка.
- **Хранилище:** SQLite (кэш данных + результаты прогонов).
- **Позже:** FastAPI + простой веб-UI для просмотра сгенерированных билдов.

---

## 11. Фазы (MVP -> рост)

- **Phase 0 — Фундамент (самый рискованный, делаем первым):**
  загрузка данных + build-model + PoB-сериализатор + headless round-trip
  (import-code -> статы). Доказать, что мы можем программно получить цифры из PoB.
- **Phase 1 — Реализатор для ОДНОГО семейства архетипов** (например, один спелл-кастер):
  получить конкретный валидный билд, который PoB принимает и оценивает.
- **Phase 2 — Энумерация скелетов + новизна** (poe.ninja baseline).
- **Phase 3 — Цикл оптимизации.**
- **Phase 4 — Масштабирование на много архетипов + UI.**

---

## 12. Риски

1. **PoB headless** — надёжность/перф. Митигация: кэш, батчи, ранний прототип на Phase 0.
2. **Реалистичность шмота** — риск «PoB-warrior» сборок, нереализуемых в игре. Митигация:
   аффиксы только из реальных пулов + ilvl + правила инфлюенсов; опц. проверка цены через ninja.
3. **Качество поиска по дереву** — эвристика Steiner-tree требует тюнинга.
4. **poe.ninja API** — лимиты/стабильность; кэшировать снапшоты.
5. **Привязка к версии патча** — данные и баланс меняются; фиксировать версию данных.

## 13. Принятые решения и открытые вопросы

Решено:
- Версия игры: **PoE 1**.
- Цель: **готовые жизнеспособные билды** (не концепты).
- **Подход: генерация с нуля** (НЕ seed-and-mutate). Реализатор строит дерево+шмот+гемы сам.
- **Амбиция цели: «поиск новой меты»** (осознанно, с принятием рисков из раздела 14).
- Целевой контент: **league-start / бюджет** (см. противоречие и его разрешение в 14.G).
- Интеграция PoB: **гибрид** (XML-контракт + долгоживущий headless luajit-процесс).
- Реалистичность шмота: **Craftable** (реальные пулы модов).
- Источник данных: **data-экспорт PoB** как единый источник.
- LLM-слой: **в контуре** (генерация гипотез + валидация data/PoB).

Открыто (для проработки):
1. Точка старта Phase 1 в пространстве осей (раздел 4.1): какой набор координат
   (способ доставки + хит/DoT + тип урона + эйлмент + крит)?
2. Метрика новизны: по `(skill × ascendancy)` или глубже (связки уников/кейстоунов)?
3. Как именно балансируем DPS vs EHP в ранжировании (Парето-фронт vs взвешенный скор)?

## 14. Требования, вытекающие из критики (обязательны для пути «с нуля + новая мета»)

Поскольку выбран самый сложный путь (генерация с нуля, амбиция «новая мета»), перечисленные
ниже проблемы НЕ обходятся, а становятся жёсткими требованиями к реализации.

### A. PoB-интеграция — защита от «тихих» ошибок
- **Контракт-валидатор:** после сериализации в PoB-формат обязательно читать обратно, какие
  моды PoB РАСПОЗНАЛ. Любой нераспознанный мод предмета = ошибка сборки, билд отбраковывается
  (а не «молча 0 урона»). Без этого DPS недостоверен.
- **Пин версии:** стаб headless и формат привязаны к конкретной версии PoB; апдейт = отдельная
  задача с регрессионными тестами на эталонных билдах (известный код -> известные цифры).

### B. Честный конфиг боя (иначе DPS завышен)
- **Профиль конфигурации обязателен и единый для всех кандидатов:** фикс уровня/типа врага,
  и реалистичные допущения по баффам/чарджам/эйлментам/аптайму фласок.
- Допущения, которые билд сам себе не обеспечивает (например «враг шокнут»), включать ТОЛЬКО
  если в сборке есть источник этого эффекта. Иначе — выключено.

### C. Модель резервации и сустейна ресурсов
- Резервация маны/жизни — жёсткий констрейнт. Реализатор обязан считать суммарную резервацию
  (с учётом эффективности/Enlighten) и сустейн каста/атаки. Нарушение = невалидный билд.

### D. Защита как профиль, а не скаляр
- Вместо одного «EHP» — вектор защит: макс-резисты, suppression, armour/evasion/block,
  имба к эйлментам, восстановление, защита от стана, chaos res.
- Жёсткие пороги (например, suppression-кап, капнутые резисты, базовая имба к ailment) — это
  констрейнты допуска; сверх них — оптимизируем.

### E. Корректные модели скейлинга (не теги)
- Отдельные математические модели: hit, DoT-скилл (RF/ED), ailment-DoT (ignite/poison/bleed),
  с учётом `more` vs `increased`, порядка конверсии, `added as`.
- Теги используются только для грубого отбора кандидатов, НЕ для оценки силы.

### F. Легальность гемов и база уников
- Модель легальности сапортов (теги, взаимоисключения, ограничения на тип скилла).
- Отдельная **база уников** (статы вне craftable-пула). Реализатор помечает «билд включён
  уником X» — это и есть основной канал новизны.

### G. Разрешение противоречия «craftable vs league-start»
- league-start = экономика, которую мы хотели отложить. Разрешение: на MVP трактуем
  «league-start» как **бюджетный профиль доступности** = «обычные/дешёвые базы + craftable
  моды среднего тира + только дешёвые уники», БЕЗ полноценной ценовой модели ninja.
- Полноценный ценовой слой (реальные цены, стоимость крафта) — отдельная поздняя фаза.
  До неё «бюджетность» = эвристический фильтр, а не точная экономика.

### H. Бюджет вычислений (перф)
- Заранее оценить: целевая пропускная способность (билдов/час), потолок PoB (~десятки мс/прогон).
- Поиск обязан быть «PoB-экономным»: грубая отбраковка дешёвыми эвристиками ДО прогона PoB;
  PoB — только для прошедших фильтр кандидатов. Кэш+дедуп идентичных сборок.

### I. Честность метрики новизны (анти-мусор)
- Низкая популярность по ninja НЕ равна «недооткрыто» — чаще «слабо». Поэтому novelty
  применяется ТОЛЬКО к кандидатам, уже прошедшим планку жизнеспособности (D+E+B).
- Явно фиксируем ограничение: на текущих данных находим «недооценённые жизнеспособные связки»;
  «новая мета» от будущих лиг/предметов — вне досягаемости до их появления в данных/PoB.

### J. Реализатор — фаза, а не модуль
- Признаём, что реализатор-с-нуля = самая большая часть работы. На MVP он жёстко сужается
  до ОДНОЙ точки осей (раздел 4.1) и небольшого набора шаблонов, иначе проект не доедет.
